# Lumo deploy verification checklist

Use this when the LMS is pointed at `https://lumo-api-production-303a.up.railway.app` and you need to prove whether the problem is:

1. the LMS is pointed at the wrong backend,
2. the backend deployment is stale, or
3. `/api/v1` is being stripped by a proxy/rewrite.

## What the repo actually expects

### LMS
- `NEXT_PUBLIC_API_BASE_URL` must be the public API origin only, for example:
  - `https://lumo-api-production-303a.up.railway.app`
- The LMS then calls absolute URLs like:
  - `${NEXT_PUBLIC_API_BASE_URL}/api/v1/meta`
  - `${NEXT_PUBLIC_API_BASE_URL}/api/v1/assets`
  - `${NEXT_PUBLIC_API_BASE_URL}/api/v1/admin/config/audit`
- There is no Next.js rewrite/proxy in `apps/lms-web/next.config.js` or root `vercel.json` that maps `/api/*` for you.
- Conclusion: the deployed LMS must point directly at an API origin that already serves `/api/v1/*`.

### API
- Health/runtime surfaces are mounted at the root:
  - `/health`
  - `/readyz`
- Product/admin routes are mounted under `/api/v1/*`, including:
  - `/api/v1/meta`
  - `/api/v1/assets`
  - `/api/v1/admin/config/audit`
  - `/api/v1/admin/assets/runtime`
- Asset URLs generated by the API use `LUMO_PUBLIC_API_URL || API_BASE_URL` as the canonical public origin.
- Conclusion: if `API_BASE_URL` / `LUMO_PUBLIC_API_URL` is wrong, runtime file URLs and config surfaces can lie even when the LMS target is correct.

## Fast diagnosis matrix

### Case A — LMS points at the wrong backend
Signals:
- `GET /health` works on the target host.
- `GET /api/v1/meta` may work.
- newer endpoints like `/readyz`, `/api/v1/assets`, `/api/v1/admin/config/audit`, `/api/v1/admin/assets/runtime` 404.
- the host responds like some older Lumo API, not the build in this repo.

Interpretation:
- `NEXT_PUBLIC_API_BASE_URL` is syntactically valid, but it points to an older or different Railway service than the one expected by the current LMS build.

### Case B — backend deployment is stale
Signals:
- old routes such as `/health` and `/api/v1/meta` work.
- new repo-backed runtime endpoints 404:
  - `/readyz`
  - `/api/v1/assets`
  - `/api/v1/admin/config/audit`
  - `/api/v1/admin/assets/runtime`
- source code clearly mounts those routes in `services/api/src/main.js`.

Interpretation:
- Railway is serving an older API release.
- This is the most likely diagnosis when some routes work but all newer ones 404.

### Case C — `/api/v1` is being stripped or rewritten
Signals:
- `GET /api/v1/assets` returns 404.
- `GET /assets` unexpectedly works, or the platform/router logs show upstream requests arriving as `/assets` instead of `/api/v1/assets`.
- same for admin/runtime routes.

Interpretation:
- an edge proxy, platform rule, CDN rule, or app rewrite is mangling the path before Express sees it.

Important:
- If both `/api/v1/assets` and `/assets` 404 on the public host, that does **not** prove prefix stripping. It points more strongly to a stale or wrong backend.

## Copy-paste verification commands

Set the target first:

```bash
export API=https://lumo-api-production-303a.up.railway.app
```

### 1) Prove what the LMS should be calling

```bash
grep -RIn "NEXT_PUBLIC_API_BASE_URL\|/api/v1/meta\|/api/v1/assets\|/api/v1/admin/config/audit\|/api/v1/admin/assets/runtime" apps/lms-web/lib apps/lms-web/app apps/lms-web/components
```

Expected:
- absolute fetches against `${NEXT_PUBLIC_API_BASE_URL}/api/v1/...`
- no dependency on Vercel rewrites for API routing

### 2) Check the live Railway host directly

```bash
curl -i "$API/health"
curl -i "$API/readyz"
curl -i "$API/api/v1/meta" -H 'x-lumo-role: admin' -H 'x-lumo-user: Pilot Admin'
curl -i "$API/api/v1/assets" -H 'x-lumo-role: admin' -H 'x-lumo-user: Pilot Admin'
curl -i "$API/api/v1/admin/config/audit" -H 'x-lumo-role: admin' -H 'x-lumo-user: Pilot Admin'
curl -i "$API/api/v1/admin/assets/runtime" -H 'x-lumo-role: admin' -H 'x-lumo-user: Pilot Admin'
```

Healthy current-build expectation:
- `/health` => `200`
- `/readyz` => `200` or `503`, but route must exist and return JSON
- `/api/v1/meta` => `200`
- `/api/v1/assets` => `200`
- `/api/v1/admin/config/audit` => `200`
- `/api/v1/admin/assets/runtime` => `200`

### 3) Specifically test prefix stripping

```bash
curl -i "$API/api/v1/assets" -H 'x-lumo-role: admin' -H 'x-lumo-user: Pilot Admin'
curl -i "$API/assets" -H 'x-lumo-role: admin' -H 'x-lumo-user: Pilot Admin'

curl -i "$API/api/v1/admin/config/audit" -H 'x-lumo-role: admin' -H 'x-lumo-user: Pilot Admin'
curl -i "$API/admin/config/audit" -H 'x-lumo-role: admin' -H 'x-lumo-user: Pilot Admin'
```

Interpretation:
- `/api/v1/...` fails but stripped-path variant works => path rewriting is happening.
- both fail => stale/wrong backend is more likely.

### 4) Check what the API itself thinks its public origin is

Run this against the real API deployment environment or logs.

```bash
echo "$API_BASE_URL"
echo "$LUMO_PUBLIC_API_URL"
```

Expected:
- one of them should equal the public Railway origin, for example:
  - `https://lumo-api-production-303a.up.railway.app`

Why this matters:
- these values drive generated asset/media URLs and config audit signals.
- the LMS can be pointed correctly while the API still emits bad public links.

### 5) Check for stale build evidence in Railway logs

Expected current build behavior from `services/api/src/main.js` startup:

```text
[lumo-api] asset routes: 4/4 mounted (...)
```

If Railway logs do not show route-evidence output or the service cannot answer `/readyz`, `/api/v1/assets`, and `/api/v1/admin/config/audit`, assume stale deploy until proven otherwise.

## Vercel-side checks

### Check the actual LMS env value used at build time
In Vercel project settings for the LMS:
- verify `NEXT_PUBLIC_API_BASE_URL`
- make sure it is exactly the public API origin, not:
  - a local URL
  - a preview URL for a different service
  - a host that already includes `/api/v1`

Correct:
- `https://lumo-api-production-303a.up.railway.app`

Wrong:
- `https://lumo-api-production-303a.up.railway.app/api/v1`
- `http://localhost:4000`
- an unrelated Railway service URL

### Confirm there is no hidden rewrite assumption
This repo currently does **not** define a Vercel/Next rewrite that proxies `/api/*` to Railway.
If someone added one in platform config outside git, remove or audit it before trusting results.

## One-line operator conclusion

If the LMS uses `NEXT_PUBLIC_API_BASE_URL=https://lumo-api-production-303a.up.railway.app`, then the current incident is most likely **wrong/stale API behind that host**, not a bad LMS fetch pattern. The LMS already calls the correct `/api/v1/*` paths; the public Railway target is the piece that does not match repo reality.
