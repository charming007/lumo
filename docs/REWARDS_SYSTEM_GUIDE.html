Rewards System Guide

A practical guide to how Lumo rewards work today across the backend/LMS and the learner tablet, what is canonical, what is provisional, and how operators should handle mismatches without bullshitting themselves.

Section 1

What is true today

Today, rewards exist in two places:

Backend / LMS reward system
Intended source of truth for reporting, leaderboard, badges, requests, and admin adjustments.
Learner tablet local reward logic
Gives immediate celebration and progress feedback on-device, especially offline, but does not fully match backend rules yet.
Practical rule: treat the backend/LMS as canonical and the tablet as provisional learner-facing encouragement.
Section 2

Current backend reward model

Backend reward logic lives in services/api/src/rewards.js.

Point earning

  • 12 XP — lesson completed
  • +3 XP — review is onTrack
  • +2 XP — no support actions used
  • +2 XP — at least one observation captured

Typical backend range per lesson: 12–19 XP

Levels / stages

LevelNameXP floor
1Starter0 XP
2Explorer40 XP
3Builder90 XP
4Navigator150 XP
5Shining Star230 XP

Badges

  • First Light — first completed lesson
  • Practice Streak — 5 completed lessons
  • Reading Spark — first English lesson
  • Math Mover — first Math lesson completed with onTrack review

Reward store / redemption flow

  • Story Time Pick
  • Sticker Reward
  • Helper Star
  • Line Leader Turn
  • Math Champion Sticker
  • Celebration Song Pick
  • Mallam Helper
  1. learner has enough XP
  2. learner requests reward
  3. admin/mallam approves or rejects
  4. approved reward is fulfilled
  5. fulfillment creates a negative XP redemption transaction
Important: XP is spendable in the backend model because fulfillment subtracts XP.
Section 3

Current learner-tablet reward behavior

Tablet reward logic currently lives in apps/learner-tablet/lib/app_state.dart.

  • gives immediate celebration after lesson completion
  • keeps reward/progress state usable while offline
  • queues sync events until connectivity returns

Local point earning

  • 12 XP — base completion
  • + up to 4 XP — based on number of responses
  • +3 XP — no support actions used
  • +2 XP — audio captured
  • +2 XP — offline / pending-sync resilience bonus

Typical tablet range: can differ from backend, and can exceed backend totals.

Section 4

Sync to LMS

  • updated runtime session data
  • updated learner progress
  • updated reward snapshot

LMS reward endpoints include /api/v1/rewards/catalog, /api/v1/rewards/leaderboard, /api/v1/rewards/requests, and /api/v1/reports/rewards.

Operator rule: Use LMS/backend rewards as truth. Use tablet rewards as immediate learner-facing feedback. If they disagree, the LMS wins.
Section 5

Manual admin adjustments

  • add manual XP
  • add a manual badge award
  • correct a reward transaction to a new XP value
  • revoke a reward transaction
Use manual adjustment to fix reward accounting problems. Do not use it to fake mastery, progression, or curriculum readiness.
Section 6

Interim operating model

  • Canonical points / levels / badges: backend
  • Immediate celebration on tablet: local tablet projection
  • Canonical spendable balance: backend/LMS

Until this area is properly unified, the contract is simple: backend truth, tablet encouragement, manual admin recovery when needed.
