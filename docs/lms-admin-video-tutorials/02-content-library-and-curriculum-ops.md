# Video 2 — Content Library and curriculum operations

**Audience:** content ops, curriculum teams, admins  
**Target runtime:** 11-13 minutes  
**Goal:** teach admins how to structure curriculum properly in Content Library and avoid fake readiness.

## Outcome

By the end of this video, the viewer should be able to:

- explain the subject → strand → module → lesson → assessment hierarchy
- create or verify curriculum structure in the right order
- understand lesson and module statuses
- use release blockers before assigning content

## Run of show

### Segment 1 — Why Content Library matters
**Narration:**
> Content Library is where the curriculum becomes structurally real. This is not just a list of content items. It is the board that tells the LMS what belongs where and what is actually safe to release.

### Segment 2 — The hierarchy
**Narration:**
> The hierarchy is subject, then strand, then module, then lesson, then assessment gate. That order matters. If lessons are created without a clean module and assessment context, the downstream delivery flow gets muddy fast.

### Segment 3 — Subject and strand lanes
**Narration:**
> Start by confirming the subject exists. Then check that the strands reflect the real planning lanes. Do not dump unrelated modules into one flat pile just because it’s faster in the moment. That kind of laziness becomes a reporting and assignment problem later.

### Segment 4 — Module cards and statuses
**Narration:**
> Each module should show level, planned lesson count, ready lesson count, and status. Draft means not release-safe. Review means almost ready. Published means it is considered live-ready. But published only means something if the lesson count and gate setup are real.

### Segment 5 — Lesson entries and quick creation
**Narration:**
> The quick lesson creation form is fine when you need to create the lesson record in the right module. But if the lesson needs objectives, step flow, media, assessment items, or interactivity, move into Lesson Studio instead of pretending metadata is enough.

### Segment 6 — Assessment gates
**Narration:**
> Assessment gates are not decoration. They control progression safety. Every serious module should have a linked gate so progression is based on evidence, not vibes.

### Segment 7 — Release blockers
**Narration:**
> Before treating a module as pod-ready, use the release blockers section. Ask four questions: do we have enough lessons, are they approved or published, is the assessment gate linked, and is the module still stuck in draft? If any of those fail, the module is not actually ready.

### Segment 8 — Inventory views
**Narration:**
> Use the curriculum release tracker for module-level scanning and the lesson inventory for lesson-level auditing. These views are how content ops stays sane when the board gets dense.

### Segment 9 — Common mistakes
**Narration:**
> The common mistakes are predictable: publishing too early, skipping the assessment gate, leaving module structure messy, and using lesson status as a bluff. The UI can help, but it cannot save bad admin judgment.

## Demo script for clicks

1. Open **Content Library**.
2. Explain the top summary cards.
3. Point at one **subject** and describe its counts.
4. Move into one **strand** and show how modules sit under it.
5. Highlight one **module** and explain level, planned lesson count, and status.
6. Point at the **lesson list** and explain draft, review, approved, published.
7. Point at the **assessment gate** panel.
8. Scroll to **Release blockers** and explain what blocks release.
9. Scroll to **Curriculum release tracker** and **Lesson inventory**.
10. End with a reminder that assignment should come after structural readiness, not before.

## Presenter notes

- Emphasize that status labels are only credible when the structure supports them.
- Call out that quick-create is not full authoring.
