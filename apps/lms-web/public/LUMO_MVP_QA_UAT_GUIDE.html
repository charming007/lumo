MVP QA / UAT Guide

An accessible, practical test guide for validating the current Lumo MVP across the LMS and learner tablet app, with real tasks, expected outcomes, pass/fail criteria, troubleshooting notes, and a clean issue capture format.

1. Purpose and testing posture

Test the MVP like it matters

This guide is for real validation, not ceremonial clicking. Use it to prove that the LMS, lesson authoring, assignments, learner runtime, and downstream evidence all behave like one coherent MVP instead of a loose pile of screens.

LMS coverage Learner app coverage End-to-end checks Release blockers called out
Recommended posture
  • Run at least one clean content-to-learner-to-evidence flow first.
  • Capture screenshots, route names, and exact steps.
  • Rate issues as blocker, major, or minor.
Do not fake confidence
  • If smoke test fails, stop and fix the blocker first.
  • If LMS and learner app disagree, treat that as a system issue.
  • If a route goes blank, that is not a “small polish bug.”

3. Test environment checklist

Before you start

  • LMS is reachable in browser.
  • Learner app build is installed and launches.
  • Backend / seed data is available.
  • At least one active subject, strand, module, and lesson exists.
  • At least one mallam, pod, and learner record exists.
  • If testing rewards, a visible learner progression path exists.
  • If testing sync recovery, know whether you are using live or cached data.
  • Know which browser, device, and build you are testing.
  • Have a bug capture template ready.
4. Fast smoke test

The 10-step “is this release even alive?” check

  1. Open LMS dashboard.
  2. Open Content Library and confirm subject → strand → module → lesson renders.
  3. Create or edit a lesson without the screen going blank.
  4. Confirm the lesson appears in the right module.
  5. Open English Studio and confirm authoring context loads.
  6. Open learner app and select a learner.
  7. Confirm expected subjects appear and no ghost content is shown.
  8. Open an assigned lesson.
  9. Complete one guided or interactive activity.
  10. Confirm completion / progress updates appear somewhere believable.
Rule: if this smoke test fails, the release is not healthy. Fix the blocker before running deeper UAT.
5. LMS test scenarios
5.1 Dashboard

Dashboard and operational visibility

TaskExpected outcomePass / Fail trigger

Fail: blank page, hard error, nonsense metrics, or broken route links.

Curriculum structure management

TaskExpected outcomePass / Fail trigger

Fail: content disappears, duplicates, lands in the wrong parent, or triggers a blank screen.

Authoring and readiness checks

Fail: blank editor, missing context, save failure, or detached lesson state.

Fail: save breaks route, structure is lost, or editor becomes unusable.

6. Learner app test scenarios
6.1 App launch

Learner selection and entry flow

  • Launch learner tablet app.
  • Select an existing learner or complete registration flow.
  • Confirm mallam / pod / learner selection does not dead-end.
Pass: learner can be selected and continue into content.
Fail: crash, blocked selection, missing options, or layout overflow that blocks progress.
6.2 Curriculum visibility

Subject and module inventory

  • Inspect visible subjects and compare with live LMS / backend expectation.
  • Open one subject and inspect modules / lessons.
  • Confirm no ghost subjects, stale demo content, or duplicate lanes appear.
Pass: learner inventory matches real curriculum.
Fail: ghost content, stale cache residue, duplicate subjects, or missing assigned material.
6.3 Lesson playback

Prompt, response, and progression flow

  • Open an assigned or available lesson.
  • Confirm prompts, instructions, and controls render.
  • Complete at least one response using speaking, listening, tapping, or choice selection.
  • Confirm the app advances or provides feedback.
Pass: learner completes meaningful interaction without being trapped.
Fail: lesson cannot open, controls break, or progression blocks mid-flow.
6.4 Persistence and recovery

Resume after interruption

  • Start a lesson and background the app or navigate away if safe.
  • Re-open the app and confirm session recovery is sensible.
  • Complete the lesson and confirm completion still records correctly.
Pass: recovery is controlled and understandable.
Fail: duplicate sessions, auto-resume loops, lost progress, or corrupted state after return.
7. End-to-end flows

Prove the LMS and learner app are the same product

Flow A — LMS content to learner runtime
  1. Create or edit a lesson in LMS.
  2. Confirm lesson is valid for delivery.
  3. Create or confirm assignment.
  4. Open learner app and targeted learner.
  5. Confirm lesson appears and can be completed.
Pass: authored content reaches the learner path. Fail: content stays trapped or learner sees something else.
Flow B — learner completion back to LMS evidence
  1. Complete a lesson in learner app.
  2. Return to LMS.
  3. Check Progress, Reports, and Rewards.
  4. Confirm at least one downstream state reflects the work.
Pass: learner activity produces believable evidence. Fail: completion vanishes or views disagree.
Flow C — interruption and recovery
  1. Interrupt a flow with refresh, route change, or app backgrounding.
  2. Return to the prior context.
  3. Confirm state recovery is clear and not corrupted.
Pass: recovery feels controlled. Fail: blank pages, duplicates, or misleading fake-success states.
8. Pass / fail model

Rate defects honestly

Blocker
  • Blank page or route crash
  • Learner cannot launch or open lessons
  • Authored content cannot be saved or reopened
  • LMS and learner app disconnect on core flow
  • Visible data corruption or impossible state
Major
  • Core flow works but is unreliable
  • Workflow confusion causes serious operator risk
  • Progress / reward / report evidence is inconsistent
  • Workaround exists but is not acceptable for pilot trust
Minor
  • Copy or styling issue
  • Small non-critical UX rough edge
  • Issue does not block core MVP completion
9. Troubleshooting notes

When things look wrong, check the boring-but-real causes first

If learner app shows ghost or stale content

  • Check local cached learner state.
  • Check fallback / seed data.
  • Check whether deprecated demo subjects or modules persist locally.
  • Check whether assignment or bootstrap data was refreshed.

If LMS content is missing from learner app

  • Check lesson status / publish readiness.
  • Check assignment existence and learner targeting.
  • Check backend sync availability.
  • Check stale app cache or offline persistence.

If LMS editor pages go blank

  • Check whether create / edit route returned the saved lesson id correctly.
  • Check required lesson data shape.
  • Check whether modal or layout overlays masked the page.
  • Check browser console for route or payload errors.

If reports or rewards look wrong

  • Check whether learner completion actually persisted.
  • Check whether sync is delayed.
  • Check whether correction / revocation logic changed totals.
  • Check whether you are comparing live data with stale expectations.

11. Sign-off checklist

Do not sign off unless the answer is honestly yes

  • Can an admin navigate the LMS without route failures?
  • Can curriculum structure be reviewed and edited safely?
  • Can an English or interactive lesson be authored and re-opened?
  • Can a learner open and complete a real lesson?
  • Does the learner app show the right content, not stale nonsense?
  • Does at least one end-to-end content-to-delivery-to-evidence flow work?
  • Are blocker and major defects either fixed or explicitly accepted?

If not, it is not sign-off. It is wishful thinking with cleaner typography.
