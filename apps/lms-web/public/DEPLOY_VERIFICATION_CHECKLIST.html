Lumo deploy verification checklist

Use this when the LMS points at https://lumo-api-production-303a.up.railway.app and you need to prove whether the problem is the LMS target, a stale backend deploy, or path mangling on the way in.

LMS should call
absolute /api/v1/* URLs

The LMS expects a plain API origin in NEXT_PUBLIC_API_BASE_URL. It should not rely on hidden rewrites.

Most likely failure
stale or wrong Railway backend

If /health and /api/v1/meta work but newer routes 404, you are probably hitting an older service, not a bad LMS fetch pattern.

Do not trust
quiet empty states

A dashboard that looks merely blank during an outage is worse than a loud blocker. Treat polite nonsense as a bug.

What this repo actually expects

  • LMS: NEXT_PUBLIC_API_BASE_URL must be the public API origin only, for example https://lumo-api-production-303a.up.railway.app.
  • LMS calls: absolute routes like ${'{'}NEXT_PUBLIC_API_BASE_URL{'}'}/api/v1/meta, /api/v1/assets, and /api/v1/admin/config/audit.
  • No hidden proxy: this repo does not define a Vercel or Next rewrite that quietly fixes bad API paths for you.
  • API runtime: health surfaces live at /health and /readyz, while product/admin routes live under /api/v1/*.
  • Generated asset URLs: come from LUMO_PUBLIC_API_URL or API_BASE_URL, so bad API env can poison links even when the LMS target is correct.

Fast diagnosis matrix

Case A — LMS points at the wrong backend
Signals: /health works and maybe /api/v1/meta works, but newer routes like /readyz, /api/v1/assets, and protected runtime audits 404.
Interpretation: the host is reachable, but it is not the current backend build this LMS expects.
Case B — backend deployment is stale
Signals: old routes work, but newer repo-backed endpoints all 404 together.
Interpretation: Railway is serving an older API release. This is the leading diagnosis when the LMS fetch pattern is already correct.
Case C — /api/v1 is being stripped
Signals: /api/v1/assets fails but /assets unexpectedly works, or platform logs show upstream requests arriving without the prefix.
Interpretation: a rewrite, proxy rule, or CDN edge layer is mangling the path before Express sees it.

Healthy current-build expectation: /health returns 200; /readyz exists and returns 200 or 503 JSON; and the runtime/admin routes above all exist instead of 404ing.
